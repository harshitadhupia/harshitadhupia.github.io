Section 01

What is Claude Code?

Claude Code is a command-line tool made by Anthropic that brings an AI coding assistant directly into your terminal. Think of it as having a senior developer sitting beside you — one who can read your entire codebase, write code, fix bugs, run tests, and explain anything, all through a conversation in your terminal.

It is not a chat window in a browser. It lives inside your actual project folder and has real access to your files. This is what makes it powerful — it sees what you see.

💻

Lives in your terminal

Works inside your project directory — not in a browser tab. It sees your actual files and folder structure.

🔍

Reads your codebase

You can ask it about any file in your project. It understands context from your entire repo, not just one snippet.

⚡

Takes real actions

It can create files, edit code, run commands, install packages, and run tests — with your permission.

🧠

Understands intent

Describe what you want in plain English. You don't need to know the exact code — just the goal.

🔄

Multi-turn conversation

It remembers your conversation context. Ask follow-up questions or say "actually, do it differently."

🛡️

You stay in control

Before running commands or making big changes, it asks for your approval. Nothing sneaks past you.

💡
Think of it this way: Claude Code = AI brain + access to your project files + terminal superpowers. It's the closest thing to pair programming with an expert who never gets tired.

Section 02

Installation — Step by Step

Claude Code installs via npm (Node Package Manager), the same tool used to install JavaScript packages. You do this once, globally, on your machine.

1

Check that Node.js is installed (version 18 or higher required)

TERMINAL
node --version # Should print v18.0.0 or higher npm --version # Should print 8.0.0 or higher

If Node is not installed, download it from nodejs.org — choose the LTS version.

2

Install Claude Code globally using npm

TERMINAL
npm install -g @anthropic-ai/claude-code

The -g flag means "global" — install it once and use it in any project folder, forever.

3

Navigate to your project folder

TERMINAL
cd /path/to/your/project # Example: cd ~/Desktop/my-app

Always run Claude Code from inside your project directory. This is how it knows what files to work with.

4

Launch Claude Code

TERMINAL
claude

On first launch, it will walk you through signing in with your Anthropic account (or creating one). After that, you're in the interactive session.

5

Check your version anytime

TERMINAL
claude --version

Keep it updated regularly to get the latest features and bug fixes.

⚠️
Mac/Linux note: If you see a "permission denied" error during npm install, prefix the command with sudo — it will ask for your system password. On Windows, run your terminal as Administrator.

Section 03

3 Ways to Use Claude Code

Claude Code isn't just a terminal tool. There are three ways to access it — and your team can pick whichever fits their comfort level best. They all use the same AI underneath, support CLAUDE.md, and can read and edit your project files.

⌨️

Terminal / CLI

Type claude in your terminal inside a project folder. Full power, all features. Best for developers already comfortable in the command line.

npm install -g @anthropic-ai/claude-code
🖥️

Desktop App

A visual interface — no terminal needed. Download the app, click to grant folder access to your project, then chat. Best for team members not comfortable in terminals.

Download from claude.ai
🌐

Claude.ai Web

On claude.ai Pro/Team, upload files or add project context. Great for code review, explaining code, and generating drafts — but no file editing or command execution.

claude.ai (browser)

🖥️ How the Desktop App Works

The Desktop App is the most beginner-friendly way to start. Here is exactly how to use it:

1

Download and install the Claude Code Desktop App

Go to claude.ai, sign in with your Anthropic account, and download the Claude Code desktop application for your OS (Mac, Windows, or Linux). Install it like any normal application.

2

Open the app and grant folder access to your project

When you open Claude Code Desktop, it will ask you to select a project folder. Click the folder picker and choose the root folder of your project (same folder that contains your code, package.json, etc.). This is how Claude gets access to your files — you explicitly grant it.

3

Start chatting — same as the terminal, but visual

You'll see a chat interface. Type your request in plain English — "Fix the login bug", "Add a new API route", "Write tests for this file." Claude reads your granted folder, makes changes, and shows you what it did. You can approve or reject each action.

4

Review changes before they're applied

Just like the terminal version, the Desktop App shows you a diff view of every file change before applying it. You see exactly what lines are added or removed. You approve it, reject it, or ask for a revision.

Section 05

Essential Commands

Claude Code has two types of commands: slash commands (typed inside the session) and CLI flags (typed in the terminal before launching). Here are the ones your team will use every day.

Slash Commands (inside the chat session)

/helpShows all available commands — your built-in reference card.
/clearWipes the conversation history and starts fresh. Use when switching to a completely different task.
/compactCompresses the conversation to save context space. Useful in long sessions.
/statusShows your current usage — how much of your context window is used.
/exitExits the Claude Code session cleanly. Also works: Ctrl+C twice.
/initCreates a CLAUDE.md file for your project by analyzing the codebase (explained in Section 05).
/bugOpens a link to report a bug to Anthropic. Use when something feels broken.

CLI Flags (before launching)

claude -p "..."Run a single prompt without entering interactive mode. Great for scripts. Example: claude -p "How many JS files are here?"
claude --versionCheck your installed Claude Code version.
claude --helpSee all available CLI options and flags.
⌨️
Keyboard shortcuts inside the session: Press ↑ / ↓ arrow keys to navigate your previous prompts (like a terminal history). Press Ctrl+C once to cancel Claude's current action. Press Ctrl+C twice to exit.

Section 06

CLAUDE.md — The Project Brain

This is one of the most important features to understand. CLAUDE.md is a special Markdown file you place at the root of your project. Every time Claude Code starts a session in that folder, it automatically reads this file first.

Think of it as a briefing document for the AI. You write instructions, context, and rules in this file, and Claude follows them every single session — without you having to repeat yourself.

📍

Location

Place it at the root of your project: ./CLAUDE.md (same level as package.json or .git folder).

🔁

Auto-loaded

Claude reads it automatically at the start of every session. You never have to mention it or include it manually.

✍️

Plain Markdown

Write it like a README — plain English with headings and bullet points. No special syntax needed.

🤖

Auto-generated

Run /init inside a session and Claude will generate a starter CLAUDE.md by reading your project.

Example CLAUDE.md for an IT company app team

# Project: Customer Portal API ## Project Overview This is a Node.js REST API for our internal customer portal. It connects to a PostgreSQL database and is deployed on AWS EC2. ## Tech Stack - Runtime: Node.js 20 + Express.js - Database: PostgreSQL 15 (via Prisma ORM) - Auth: JWT tokens (see /src/middleware/auth.js) - Tests: Jest + Supertest ## Coding Rules - Always use async/await, never callbacks - All new functions MUST have JSDoc comments - Variable names must be descriptive — no single-letter vars - Error handling: always use our custom AppError class ## File Structure - /src/routes — All Express route files - /src/services — Business logic (no DB queries here) - /src/models — Prisma models - /src/middleware — Auth, error handling, validation ## How to run locally npm run dev → starts dev server on port 3000 npm test → run all Jest tests npm run migrate → run Prisma database migrations ## Important Notes - NEVER commit the .env file - The staging DB connection string is in .env.staging - Ask before running any database migration commands
🎯
Pro tip: Once you create a good CLAUDE.md, commit it to Git. Every team member who uses Claude Code on this project will automatically get the same context — consistent AI behavior across your whole team.

Section 06B

CLAUDE.md vs skills.md — What's the Difference?

Now that you understand CLAUDE.md, there's a second instruction file worth knowing: skills.md. Both give Claude rules and guidelines — but they work very differently. Understanding the difference will help your team write smarter, more efficient instructions.

📋
CLAUDE.md
Global · Always loaded

Think of it as a README for your AI. It holds overarching, permanent rules that apply to every task in the project.

→General architecture overview
→Coding standards (PEP 8, ES6, etc.)
→Package manager preferences
→Workflow rules & team conventions
→How to run, test, and deploy
⚠ Token cost
Loaded on every single command — keep it focused and reasonably short to avoid inflating costs.
🛠️
skills.md
Dynamic · Loaded only when needed

Task-specific instruction packs. Skills are folders of instructions loaded only when Claude detects they're relevant to the current task.

→How to format structured logs
→Use PyPDF whenever working with PDFs
→Native UI rules for Expo mobile apps
→Company-specific boilerplate patterns
→Domain-specific library preferences
✓ Token efficient
Loaded only when relevant — keeps the context window clean and saves API costs on unrelated tasks.
💡
Simple analogy: CLAUDE.md is like the employee handbook — everyone reads it on day one and it applies always. skills.md is like specialist training manuals — you only pull them out when you're doing that specific type of work.

How to Create Each File

📋 CLAUDE.md
Option A — Auto-generate
Run /init in a Claude Code session. It reads your project and writes the file for you.
Option B — Write manually
Create a file named CLAUDE.md at your project root and type your standards in plain Markdown.
📁 Nested files in monorepos
Large repos with sub-projects can have multiple CLAUDE.md files at different levels. Claude uses the one closest to the folder you're working in.
🛠️ skills.md
Option A — Install pre-built skills
Community skills exist for frameworks like Expo. Example:
npx skills add expo-skills
Option B — Write a custom skill
Create a Markdown file with the exact instructions and boilerplate Claude should follow for that task type (e.g., your company's log formatting rules).
📁 Where skills are stored
Usually auto-placed in .claude/ or agents/ folders. Can be saved locally (one project) or globally (all projects on your machine).

Quick Comparison

Property CLAUDE.md skills.md
Purpose Global project rules & architecture Task-specific instructions & patterns
When loaded Every single command, always Only when Claude detects it's relevant
Token cost ⚠ Always consuming ✓ Only when needed
Created via /init command or manually npx skills add or manually
Scope Project-wide (or sub-directory) Local to project or global across machine
Best for Standards, stack info, run commands Library choices, formatting, boilerplate
🎯
Practical advice for your team: Start with just CLAUDE.md — it covers 90% of what you need day-to-day. As your team identifies repetitive tasks where Claude keeps needing the same specific guidance (e.g. "always use our logging library"), that is the right moment to extract it into a skill.

Section 07

How to Write Good Prompts

The quality of what Claude Code produces depends heavily on how you ask. You don't need to be technical — but you need to be specific and clear. Here are real examples your team can use immediately.

CODE "Create a new Express route in /src/routes/users.js that handles GET /users/:id — it should fetch the user from the database using Prisma and return a 404 if not found."
DEBUG "I'm getting this error: TypeError: Cannot read properties of undefined (reading 'userId'). It happens when I call POST /orders. Find the bug and fix it."
DOCS "Read all the files in /src/routes and write a Markdown summary of every API endpoint — method, URL, what it does, and what it returns."
REFACTOR "The function in /src/services/orderService.js is 200 lines long. Refactor it into smaller functions with clear responsibilities. Keep the same behavior."
TESTS "Write Jest unit tests for /src/services/authService.js. Test the happy path, invalid credentials, and expired token scenarios."
CODE "Add input validation to the POST /users route. Validate that email is a valid email format, name is at least 2 characters, and phone is optional but must be a valid number if provided."
DEBUG "My Jest tests are failing with this output: [paste error]. Look at the test file and the implementation and tell me why they're failing."
EXPLAIN "Read /src/middleware/auth.js and explain in simple terms what it does. I'm new to JWT and don't understand how the token verification works."
📝
3 Rules for Great Prompts:

1. Be specific about the file or location — "in /src/routes/auth.js" is better than "in my auth code"
2. State the expected behavior — "return 404 if not found" gives Claude a success condition
3. Paste error messages directly — copy-paste the full error text; don't paraphrase it

Section 08

Plan Mode — Think Before You Build

Plan Mode is one of the most powerful features in Claude Code, and the one most beginners skip. Instead of immediately writing code, Plan Mode makes Claude first create a detailed written plan — which files it will touch, what it will change, and how it will verify the work — and shows it to you for review before a single line of code is written.

💡
One sentence summary: Plan Mode = Claude tells you what it's going to do, you approve the plan, then it executes. This gives you full visibility and control, especially for large or risky tasks. Think of it as a mandatory code review before the code even exists.

How to activate Plan Mode

Simply add the word "plan" or ask Claude to plan before acting — either via a slash command or in your prompt:

/planToggle Plan Mode on inside the session. Claude will plan before executing any task you give it.
"Plan this first, then wait for my approval"You can also just ask Claude in plain English to plan before doing anything.
"Before writing any code, ask me questions and show me a plan"Combines question-asking and planning in a single instruction — great for big features.

What a Plan looks like

When Claude creates a plan, it outputs something like this before touching any file:

PLAN MODE OUTPUT — example
## Plan: Add Password Reset Feature Architecture: - New route: POST /auth/forgot-password - New route: POST /auth/reset-password - New service: /src/services/passwordResetService.js - DB change: Add reset_token + expiry columns to users table - Uses: existing emailService.js for sending the email Files I will MODIFY: ✎ /src/routes/auth.js (add 2 new routes) ✎ /prisma/schema.prisma (add reset token fields) Files I will CREATE: ✚ /src/services/passwordResetService.js ✚ /prisma/migrations/add_reset_token.sql Files I will NOT touch: ✓ /src/middleware/auth.js ✓ /src/services/emailService.js (will import, not modify) Verification steps: 1. Run: npx prisma migrate dev 2. Run: npm test -- --grep "password reset" 3. Manual test: POST /auth/forgot-password with valid email Do you approve this plan? (yes / adjust first)

7 Best Practices for Plan Mode

1

Break down large features

Don't ask Claude to build a massive feature all at once. Break large requests into smaller, manageable chunks so you can review code along the way and prevent the agent from modifying too many files at once.

2

Provide rich detail + visual aids

Explain requirements as thoroughly as possible. You can paste screenshots or UI mockups directly into the terminal — Claude can see images and use them as a visual reference for what you want to build.

3

Force Claude to ask questions

In your prompt, explicitly say: "Ask me questions before you start." This creates a dialogue that ensures you and Claude are perfectly aligned on what to build before it writes a single line.

4

Set architectural boundaries

Remind Claude to keep code composable, reusable, and split across proper files. Without this, Claude may dump everything into one giant file — hard to maintain and expensive on your context window.

5

Thoroughly review the drafted plan

Once Claude generates the plan, actually read it. Check the list of files it will modify. For serious projects, this review is what saves you from unwanted changes buried in files you didn't expect to be touched.

6

Iterate before executing

If you notice something missing in the plan — say, a confirmation dialog on the delete button — don't just approve it. Ask Claude to adjust the plan first, then approve the revised version.

7

Manage context upon approval

When you finally execute the plan, you'll be given options. If the new feature is completely unrelated to previous tasks in the session, choose "clear context and auto-accept edits" — this gives Claude a fresh memory, keeps it sharp, and saves you API costs.

🏆
The ideal Plan Mode prompt template:

"I need to add [feature]. Before writing any code: ask me any clarifying questions you need, then write a plan showing which files you'll create or modify and how you'll verify it works. Wait for my approval before starting."

This single prompt activates question mode + plan mode + approval gate — the full power of Plan Mode in one instruction.

Section 08B

MCP — Connecting Claude to the Outside World

MCP (Model Context Protocol) is an architecture that lets you extend Claude Code by connecting it to external tools and services — things like GitHub, Slack, a Postgres database, or Google Drive. Without MCP, every AI provider would have to write and maintain custom code for every external platform's unique API. MCP solves this with a standardised connection layer that works the same way regardless of which service you're connecting to.

💡
One sentence summary: MCP is a plug socket standard. Any external service that builds an MCP Server becomes instantly usable by Claude Code — no custom integration code needed on your end.

How MCP Works — Two Components

🖧
MCP Server
Lives with the external service

Wraps an external platform (e.g. GitHub). It knows all the API complexity — authentication, message formats, endpoints. If GitHub changes their API, only the MCP Server needs updating — your Claude setup stays the same.

🤖
MCP Client
Lives inside Claude Code

When Claude Code starts, the client asks each registered MCP Server: "what tools do you offer?" Claude then knows exactly which tool to call when you ask it to do something — no custom code on your side needed.

How a request flows
You type a request
→
Claude (MCP Client)
→
MCP Server (e.g. GitHub)
→
GitHub API
→
Result back to you

Popular MCP Servers — Real Examples

There are hundreds of MCP servers available. Here are the most useful ones for an IT app team:

🐙

GitHub

Create issues, open pull requests, read repositories, review code — all by asking Claude in plain English.

💬

Slack

Send messages to channels, read conversations, post notifications — Claude can interact with your team chat.

🐘

Postgres / Databases

Query your database, inspect schemas, run updates — Claude can read and write data directly.

🎭

Playwright

Automate browser testing. Ask Claude to run end-to-end tests and report results without manual test scripts.

📁

Google Drive

Access, read, and reference documents from Drive — useful for grabbing specs or design docs mid-session.

🚨

Sentry

Check error logs, inspect recent exceptions, and ask Claude to help diagnose production issues directly from Sentry data.

How to Register an MCP Server

You add MCP servers to Claude Code via a config file or command. Once registered, Claude automatically discovers their tools at session start.

TERMINAL — adding an MCP server
# Add an MCP server (example: GitHub) claude mcp add github # List all registered MCP servers claude mcp list # Remove an MCP server claude mcp remove github # Once registered — just ask Claude naturally: > "Create a GitHub issue titled 'Fix login bug' in our repo" > "Check Sentry for any new errors in the last 24 hours" > "Query the users table and show me all accounts created today"
⚠️
Critical best practice — only add what you need.

Every registered MCP server loads its available tools into Claude's context window at session start. Adding too many servers you don't use will:

  • Increase your API costs — more tokens consumed per session
  • Burn through usage limits faster
  • Confuse Claude — too many tools to choose from can lead to wrong tool selection

Rule of thumb: Register only the MCP servers your team actively uses in this project. Remove ones you've stopped using.

Section 10

The Right Mindset for Using Claude Code

The biggest factor in getting great results isn't technical knowledge — it's how you think about working with Claude. Here's what separates teams that love it from those who feel it "doesn't work."

❌ Don't do this
  • Say "fix my code" without context
  • Accept every change without reading it
  • Use it as a replacement for understanding your own code
  • Expect it to guess which file you mean
  • Give up after one bad response
  • Put production secrets in prompts
✅ Do this instead
  • Name the specific file and the exact problem
  • Read and test every change before committing to Git
  • Use it to learn — ask "why did you do it this way?"
  • Provide full error messages, not summaries
  • Rephrase or add context if the first response misses
  • Treat Claude like a junior dev: verify its work
🏆
The most powerful pattern — give Claude a task and let it work:

"I need to add a feature that lets users reset their password. The email service is already set up in /src/services/emailService.js. Create the route, the service function, and the Prisma model changes needed. Run the tests when you're done."

That single message can do what would take a developer 2–3 hours. The key is: specific goal + point to existing files + define the done condition.

Section 11

FAQ — Questions Your Team Will Ask

Q

Is my code being sent to the internet? Is it private?

Yes, when you use Claude Code, the content of your files and prompts is sent to Anthropic's servers for processing. Use your company's Anthropic plan which includes data privacy agreements. Never put passwords, API keys, or production secrets directly in your prompts.

Q

What if Claude gives a wrong answer or bad code?

This happens — just like a human developer can make mistakes. Always review the code before accepting. You can say "that's wrong because X, try again" or "this breaks the existing tests, here's the error" and Claude will self-correct. Treat it like reviewing a pull request.

Q

Does it work with any programming language?

Yes. Claude Code works with JavaScript, TypeScript, Python, Java, Go, Ruby, PHP, C#, and virtually any language you'll encounter. It also works with config files, SQL, Docker, shell scripts, etc.

Q

Can multiple developers on our team use it on the same project?

Absolutely. Each developer runs their own Claude Code session locally. If you commit the CLAUDE.md file to your Git repo, everyone automatically gets the same project context — making the AI consistent across your team.

Q

What happens if the session "forgets" earlier context?

Claude Code has a context window limit. In very long sessions, it may lose track of earlier conversation. Use /compact to compress history, or /clear to start fresh. The CLAUDE.md always reloads, so project context is never truly lost.

Q

Can I use it without an internet connection?

No — Claude Code requires an internet connection to reach Anthropic's API. It cannot run fully offline.
